### Architettura delle applicazioni

- Client - Server
    
    Server - sempre attivo con IP fisso
    
    Client - fa richieste al server, IP dinamico, non comunica con altri client
    
- Peer to Peer - P2P
    
    Non esistono server, coppie di client comunicano tra di loro, non devono essere sempre attivi
    
- Ibrido
    
    Presenza di un server per la ricerca di utenti, risorse, etc… ma connessione client-client diretta

### Web e HTTP

Una pagina web è formata da un file base HTML e da altri oggetti referenziati all’interno. Ogni oggetto è referenziato da un URL (Uniform Resource Locator)

HTTP (HyperText Transfer Protocol)

- protocollo a livello applicazione del Web,
- modello client/server
- connessione TCP su porta 80
- stateless - il server non mantiene info sulle richieste del client

Tipi di connessione HTTP:

- Persistenti
    
    Più oggetti possono essere trasmessi su una singola connessione TCP
    
- Non persistenti
    
    Viene trasmesso solo un oggetto. Per ogni oggetto va stabilita una nuova connessione
    

### Connessioni non persistenti

1. Il client HTTP inizializza una connessione TCP con il server HTTP all’URL sulla porta 80
2. Il server HTTP accetta la connessione e avvisa il client
3. Il client HTTP trasmette un messaggio di richiesta di un oggetto alla socket TCP
4. Il server HTTP riceve il messaggio di richiesta, forma il messaggio di risposta e lo invia
5. Il client HTTP riceve il messaggio di risposta e lo visualizza
6. Il server HTTP chiude la connessione, se è sicuro che il client abbia ricevuto correttamente la risposta

RTT → tempo impiegato da un pacchetto per andare e tornare dal server

### Connessioni Persistenti

- Dopo la prima richiesta e mandata la prima risposta il server lascia la connessione aperta
- Ogni richiesta del client per gli oggetti viene inoltrata sulla stessa connessione
- La connessione viene chiusa quando rimane inattiva oltre un certo tempo

### Cookie

Essendo HTTP un protocollo stateless, non viene salvato alcuno storico delle interazioni effettuate dallo stesso client. Per ovviare a questo problema si usano i cookie.

- Alla prima interazione del client, il server crea un nuovo ID, lo salva nel suo database e lo manda al client con il primo messaggio di risposta
- Ad ogni interazione successiva sarà il client a specificare e mandare il suo cookie in modo da far sapere al server quale utente siamo

### Cache Web (Server Proxy)

Per rendere la visualizzazione di una risorsa più veloce si può usare un server proxy. è un server fisicamente più vicino al client che si frappone fra il client e il server. Il browser del client farà richieste al proxy invece che al server finale.

Quindi, il browser fa una richiesta al proxy, se questo si ritrova la risorsa nella memoria, la manda al client, se non l’ha, la richiede al server finale e la manda al client.

Il problema che può apparire con un proxy è che la risorsa che è memorizzata nella cache non sia aggiornata. La soluzione a questo si trova con una GET condizionale.

Il proxy fa una richiesta al server con una riga particolare nell’intestazione, *If-modified-since: data.* Se l’oggetto non è stato modificato allora il server manda una risposta vuota con codice *304 Not Modified* 

### Posta elettronica

Per funzionare servono 4 componenti, 2 agenti utenti (mittente e destinatario) e 2 server di posta (mittente e destinatario).

Agente utente → composizione, editing e lettura di messaggi

Server di posta → contiene i messaggi in arrivo e inoltra i messaggi inviati al server destinatario

### SMTP - Simple Mail Transfer Protocol

Protocollo per trasferire i messaggi da client a server con TCP porta 25. Prevede un ulteriore di fase di handshake (a livello applicativo) oltre a quella per la connessione TCP (livello trasporto)

I messaggi SMTP hanno sempre la struttura di comando/risposta. I comandi sono del testo ASCII a 7 bit, i messaggi di risposta sono formati da un codice di stato e da un eventuale espressione testuale

Esempio di interazione SMTP

```jsx
S: 220 hamburger.edu          // è il server a mandare il primo messaggio
C: HELO crepes.fr
S: 250 Hello crepes.fr, pleased to meet you     // codice di OK con messaggio impostato a piacere
C: MAIL FROM <alice@crepes.fr>           //comando mail seguito dal mittente
S: 250 alice@crepes.fr Sender ok         //comando ok se la mail viene riconosciuta
C: RCPT TO: <rob@hamburger.edu>          //comando RCPT seguito dal destinatario
S: 250 rob@hamburger.edu recipient ok    //comando ok se mittente riconosciuto
C: DATA                                  //comando per dire che sta cominciando a mandare la mail
S: 354 Enter mail, end with "." on a line by itself
C: Do you like ketchup?
C: How about pickels?
C: .
S: 250 Message accepted for delivery
C: QUIT 
S: 221 hamburger.edu closing connection 
```

SMTP usa connessioni persistenti, e più oggetti vengono trasmessi in un unico messaggio

L’RFC 5322 definisce uno standard per il formato dei messaggi di testo. Deve essere formato da delle righe di intestazione (To, From, Subject, CC…), una riga vuota e a seguire il corpo del messaggio in formato ASCII

### Protocolli di accesso alla posta

Il protocollo SMTP serve solo per inviare una mail tra un client e un server, ma per poter accedere ai messaggi salvati sul server si utilizzano protocolli diversi.

- POP3 - Post Office Protocol - prevede l’autenticazione e il download
- IMAP - Internet Mail Access Protocol - Funzioni più complesse e manipolazione dei messaggi salvati
- HTTP - Gmail, Libero…

### POP3

```jsx
//autorizzazione         in questa fase vengono mandati username e password del client
//											 il server ha solo due risposte, +OK e -ERR
S: +OK POP3 server ready       // come nell'SMTP è il server a mandare il primo messaggio dopo l'handshake TCP
C: user rob
S: +OK
C: pass hungry
S:+OK 

//fase di transazione
C: list                         //elenca tutti i messaggi presenti
S: 1 498                        // ID + dimensione in byte
S: 2 912
S: .
C: retr 1                       // richiede l'email con ID 1
S: message 1 contents           // contenuto del messaggio
S: .
C: dele 1                       //la cancellazione effettiva del messaggio avviene dopo la chiusura della conversazione
C: retr 2
S: message 2 contents
S: .
C: dele 2
C: quit
S: +OK POP3 server signing off

```

POP3 può essere usato sia in modalità scarica e cancella, come sopra, o scarica e mantieni. Inoltre non mantiene lo stato fra le varie sessioni

### DNS - Domain Name System

Protocollo e servizio usato per tradurre i nomi simbolici degli host in indirizzi IP. Si tratta sia di un database distribuito in cui vengono salvate le coppie nome-IP che di un protocollo applicativo che consente agli host di comunicare per tradurre gli indirizzi.

Servizi DNS:

- Traduzione degli hostname in IP
- Host aliasing - un host può avere più nomi
- Mail server aliasing - avere lo stesso nome sia per il server mail che per il server web
- Distribuzione locale - stesso server web replicato su diverse macchine in modo da poter distribuire il carico

I database DNS sono organizzati gerarchicamente. Esistono alcuni server DNS radice, a cui fanno capo più server DNS che si occupano dei top level domain, ossia dei .com, .org, etc… . A loro volta questi raggruppano i server DNS che si occupano dei singoli domini (autoritativi).

Quindi per esempio per arrivare ad amazon.com:

- Il client interroga il server radice per trovare il server DNS com
- Il client interroga il server DNS com per ottenere il server DNS amazon.com
- Il client interroga il server [amazon.com](http://amazon.com) per trovare l’IP di www.amazon.com

 Fuori da questa gerarchia può essere presente un server DNS locale appartenente alla LAN, chiamato anche Default Name Server, che serve a togliere il carico dal client di fare tutte le richieste e anche da cache.

### Record DNS

L’unità più piccola di memorizzazione di informazioni è l’RR (Resource Record)

Formato: (name, value, type, ttl)

TTL→ Time To Live. Tempo di vita dell’informazione. Scaduto questo muore

Name e value assumono informazioni diverse in base al valore di Type:

- Type = A
    - *Name* è il nome dell’host
    - *Value* è l’indirizzo IP
- Type = CNAME
    - *Name* è l’alias di qualche nome canonico (www.ibm.com)
    - *Value* è il nome canonico (servereast.backup2.ibm.com) **
- Type = MX (come cname ma per la posta)
    - *Value* è il nome del server di posta
    - *name* nome effettivo del server
- Type = NS (serve per sapere a chi chiedere la traduzione)
    - *Name* è il dominio
    - *Value* è il nome del server di competenza
