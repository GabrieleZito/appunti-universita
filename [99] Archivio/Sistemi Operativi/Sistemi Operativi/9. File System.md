record → gruppo di campi

File → raccolta di dati associata ad un nome, manipolabile come un’unica entità attraverso diverse operazioni (apertura, chiusura, lettura, copia…).

Volume → unità di memorizzazione di dati

FILE SYSTEM → organizza i file e gestisce gli accessi ai dati, alloca spazio sui dispositivi secondari e terziari e assicura l’integrità dei file.

Sono in grado di strutturare i file per ogni applicazione nel modo più appropriato. Inoltre un file system dovrebbe essere indipendente dal dispositivo (i.e. posso continuare a riferirmi ad un file con il suo nome logico)

Per organizzare i file si usano le cartelle (directory). Queste possono avere diverse strutture. Una è quella a livello singolo, cioè esiste una sola cartella che contiene tutti i file. Usata raramente. Un’altra struttura è quella gerarchica. Esiste una cartella radice che contiene tutte le altre. Ogni cartella, che è un file che punta ad altre cartelle o file, deve contenere al suo interno file con nomi unici.

  

Soft Link / link simbolico / scorciatoia → elemento che contiene il path-name di un altro file

Hard link → elemento che specifica la posizione del file sul dispositivo fisico

  

Metadati → posizione di blocchi liberi nel dispositivo e ora di ultima modifica di un file. Servono per proteggere l’integrità del file system.

Formattazione → Dipende dal sistema, ma in genere è l’individuazione di eventuali settori inutilizzabili, cancellazione di tutti i dati e creazione della directory radice.

Descrittori dei file → Quando si apre un file, per evitare ogni volta di riattraversare tutto il percorso, viene salvata in RAM una tabella che tiene traccia dei file aperti. Il descrittore del file è l’indice della tabella.

La tabella contiene inoltre il File Control Block, che specifica gli attributi del file (nome simbolico, posizione sul dispositivo, tipo di dispositivo, ultima modifica, creazione…)

Montaggio → unificare più file system in un unico namespace. Assegna una directory del file system nativo, detta punto di montaggio, alla radice del file system montato

  

### Organizzazione dei file

Modo in cui i record di un file sono sistemati sul dispositivo secondario.

- Sequenziale → Un record è posizionato fisicamente successivamente ad un altro.
- Diretta → I record sono referenziati direttamente tramite l’indirizzo fisico sul dispositivo di memoria.
- Sequenziale indicizzata → Gruppi di record sequenziali accessibili attraverso le chiavi contenute in ogni record, oppure direttamente
- Partizionata → File di sottofile sequenziali

### Allocazione dei file

- Allocazione contigua → i file vengono posizionati in indirizzi contigui. L’utente specifica prima la dimensione del file, se non c’è spazio contiguo disponibile, il file non viene creato. Il vantaggio è che di solito i file cercati successivamente saranno probabilmente lì vicino. Svantaggio la frammentazione
- Allocazione con liste collegate non contigue → Ogni elemento delle directory punta al primo settore di un file su di un dispositivo. Svantaggio che i settori dei file potrebbero essere molto dispersi.
- Allocazione non contigua tabellare → Una tabella memorizza i puntatori ai blocchi in modo contiguo. Per rendere la ricerca veloce la tabella andrebbe salvata sul disco in modo contiguo e collocata nella cache nella RAM.
- Allocazione indicizzata non contigua → Ogni file possiede uno o più blocchi indice. Questo contiene un elenco di puntatori ai blocchi di dati del file. Più blocchi indice vengono collegati tra loro. Il vantaggio è che la ricerca può limitarsi solo ai blocchi indice, che vengono tenuti vicini. è possibile anche organizzarli gerarchicamente con blocchi indice di blocchi indice.

### Gestione dello spazio libero

- Lista di blocchi collegati → Puntatori ai blocchi liberi. Si aggiungono i blocchi appena liberati in coda e si allocano i blocchi in testa. un problema è che lo spazio in memoria può diventare frammentato, quindi blocchi contigui nella lista non lo saranno sul disco
- Bitmap → Un bit per ogni blocco del file system. l’i-esimo bit è posto a 1 se il blocco i-esimo corrispondente è occupato. Vantaggio, si possono trovare facilmente i blocchi contigui, svantaggio, a volte potrebbe essere necessario cercare per tutta la bitmap.

### Controllo di accesso ai file

- Matrice di controllo degli accessi → Matrice con tutti gli utenti e file del sistema. l’elemento $a_{ij}$﻿ è 1 se l’utente $i$﻿ può accedere al file $j$﻿. Una matrice del genere occuperebbe un sacco di spazio. E sarebbe più utile specificare anche i diversi tipi di accesso.
- Classi di utenti → Gli utenti vengono categorizzati in diverse categorie, proprietario, utente specifico, gruppo e pubblico. La categoria viene salvata nel file control block.

### Tecniche di accesso ai dati

- Metodi di accesso a coda → usati quando può essere prevista la sequenza in cui sono stati processati i record (accessi sequenziali e seq. indicizzati). Cercano quindi di rendere disponibile il record successivo ancora prima che sia elaborato quello precedente.
- Metodi di accesso di base → usati quando la sequenza non è prevedibile (accesso diretto). Oppure nei casi in cui l’utente vuole il controllo dell’accesso ai record senza il sovraccarico della previsione.

Quando un processo fa una richiesta di scrittura, la pagina viene caricata in memoria principale e la pagina corrispondente viene segnata come _dirty_ . Quando il file viene chiuso, le pagine dirty vengono copiate sulla memoria secondaria

### Backup e recupero

Per una protezione fisica dei dati, usare un UPS per mantenere acceso il sistema, fare un backup in un luogo geograficamente distante.

Due tipi di backup

- Backup fisico → duplicazione dei dati a livello di bit. Non memorizzano la struttura logica del file system, quindi non distinguono i file che contengono. Si dovrebbe memorizzare quindi l’intero file system, duplicando tutti i dati ogni volta, anche se non sono stati modificati dall’ultimo backup.
- Backup logico → memorizza i dati e la struttura logica del file system. Conoscendo la struttura, quindi possono anche determinare quali file abbiano bisogno di backup. Inoltre i backup logici sono interpretabili anche da sistemi con architetture diverse. In un backup logico possono non essere inclusi alcuni file, come i file nascosti o i metadati.
- Backup incrementale → Backup logico che memorizza solo i file modificati rispetto all’ultimo backup. Molto veloce.

### Logging e Paginazione shadow

Per evitare che un guasto durante un’operazione lasci i dati corrotti, si usa il logging delle transazioni. Le transazioni sono gruppi di operazioni atomiche, cioè o vengono eseguite tutte o nessuna. Le operazioni di una transazione vengono eseguite e i risultati vengono prima salvati in un log. Effettuata la transazione, viene confermata registrando un valore speciale nel log. In un momento successivo vengono apportate le modifiche anche nella memoria permanete. Mantenendo salvati tutti i log è possibile annullare un qualsiasi numero di operazioni effettuate.

Con la paginazione shadow le operazioni vengono effettuate, ma salvando i dati in un blocco libero, diverso da quello originale. Se la transazione va a buon fine il blocco nuovo diventa quello originale e quello vecchio viene liberato

  

### Log File System

Eseguono le operazioni di file system come transazioni registrate. Es creazione di un file in un cartella. Si scrivono i dati del file nel log, si scrivono i metadati e infine si scrive il nuovo file nella cartella. La scrittura di file avviene solo alla fine, quindi per cercare un file specifico si potrebbe dover leggere tutto il log. Per evitarlo a volte si scrivono nel log solo i metadati

  

### File server

gestione non locale dei riferimenti ai file. Può essere un collo di bottiglia, visto che più client devono fargli richieste. è meglio quindi un Network File System, in cui ogni computer ha un file system che può fare sia da client che da server. Questo tipo di file system distribuito permette di modificare file remoti come se fossero locali, anche se tra due macchine che architetture diverse.

  

### Database

Raccolta dati integrata e controllata centralmente. Vantaggio che riduce la ridondanza dei file uguali, ma che hanno solo dei pathname diversi. I database hanno l’indipendenza dei dati. le applicazioni cioè non hanno accesso diretto ai dati e alla struttura organizzativa. Più applicazioni possono vedere gli stessi dati ma con viste logiche diverse.

Linguaggi per database

- DDL data definition language → specifica come sono organizzati i dati e come si relazionano tra loro
- DML data manipulation language → modifica dei dati
- QL query language

Database distribuito → database sparso su più macchine in una rete
