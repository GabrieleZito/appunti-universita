Più thread in un sistema che esistono allo stesso momento si dicono concorrenti. Se agiscono in maniera indipendente, ma possono cooperare e sincronizzarsi si dicono asincroni. Quando un thread accede ad una variabile, questa deve essere aggiornata e prova di errori. Quindi un sono thread alla volta dovrebbe poter accedere alla variabile. Questo è il problema della mutua esclusione.

Quando più thread leggono solo la stessa variabile non c’è problema, ma quando un thread cerca di leggere mentre un altro thread sta modificando, si può arrivare a risultati indeterminati. Per risolvere il problema, ogni thread deve avere un accesso esclusivo alla variabile. Deve esserci quindi una lista di attesa per accedere alla variabile. Questo ha il nome di serializzazione dell’accesso alla variabile condivisa.

Se prendiamo come esempio due thread, che lavorano sulla stessa variabile, uno scrive e uno legge, può capitare che la variabile venga sovrascritta più volte senza che venga prima letta, o che venga letto più volte lo stesso valore. Dobbiamo dividere il codice in sezioni logiche, e quelle in cui si lavora sulle variabili sono sezioni critiche. Per gestire l’accesso alle sezioni critiche vengono definite due primitive: Wait e Notify. La prima indica che il thread sta aspettando che la variabile si liberi per l’utilizzo, la seconda che il thread che stava usando la variabile ha finito e che la sta liberando.

Esistono diversi algoritmi per implementare la mutua esclusione:

- Algoritmo di Dekker (2 thread)
    
    Usa un flag per indicare che un thread vuole entrare nella sezione critica, ma c’è un favorito in caso di pareggio
    
    ```Java
    int favoredThread = 1
    boolean t1WantsToEnter = false
    boolean t2WantsToEnter = false
    startThreads()
    
    void main(){           //Thread 1
    	while(!done){
    		t1WantsToEnter = true
    		while(t2WantsToEnter){
    			if(favoredThread==2){
    				t1WantsToEnter = false
    				while(favoredThread==2);
    				t1WantsToEnter = true
    			}
    		}
    		//Sezione critica
    		favoredThread = 2
    		t1WantsToEnter = false
    		//Fine sez critica
    	}
    }
    
    void main(){           //Thread 2
    	while(!done){
    		t2WantsToEnter = true
    		while(t1WantsToEnter){
    			if(favoredThread==1){
    				t2WantsToEnter = false
    				while(favoredThread==1);
    				t2WantsToEnter = true
    			}
    		}
    		//Sezione critica
    		favoredThread = 1
    		t2WantsToEnter = false
    		//Fine sez critica
    	}
    }
    ```
    
- Algoritmo di Peterson (2 thread)
    
    Come per Dekker, ma il thread favorito si alterna
    
    ```Java
    int favoredThread = 1
    boolean t1WantsToEnter = false
    boolean t2WantsToEnter = false
    startThreads()
    
    void main(){            //Thread 1
    	while(!done){
    		t1WantsToEnter = true
    		favoredThread = 2
    		while(t2WantsToEnter && favoredThread == 2);
    		//Sezione critica
    		t1WantsToEnter = false
    		//Fine sezione critica
    	}
    }
    
    void main(){            //Thread 2
    	while(!done){
    		t2WantsToEnter = true
    		favoredThread = 1
    		while(t1WantsToEnter && favoredThread == 1);
    		//Sezione critica
    		t2WantsToEnter = false
    		//Fine sezione critica
    	}
    }
    ```
    
- Algoritmo di Bakery di Lamport (N thread)
    
    Mutua esclusione tra n thread usando una sola variabile condivisa. è modellato seguendo l’esempio di un panificio che deve servire più clienti. Se c’è un solo Thread, questo viene servito normalmente. Se sono presenti più thread, c’è bisogno di un modo di tenere l’ordine in cui hanno fatto richiesta. Per farlo si usa un array di “biglietti” numerati. Quando un thread possiede il biglietto con il valore più basso, allora può entrare nella sezione critica. Ogni Thread prende un biglietto impostando il valore come ticket[x] = maxValue(ticket) +1. Prima che un thread possa entrare nella sezione critica bisogna ovviamente controllare che sia il suo turno, e quindi che il ticket[x] del thread abbia il valore più basso. Finita la sezione critica il ticket del thread viene azzerato. Un altro controllo importante da fare è che solo un Thread alla volta prendano il biglietto, poichè l’operazione maxValue() potrebbe richiedere del tempo, e il thread sospeso prima che possa finire, con il rischio di non rispettare la mutua esclusione
    
    ```Java
    boolean choosing[n]
    int ticket[n]
    
    startThreads():
    
    void main(){
    	x = threadNumber()
    	while(!done){
    		choosing[x] = true
    		ticket[x] = maxValue(ticket) + 1
    		choosing[x] = false
    
    		for(int i = 0; i<n; i++){
    			if(i==x){
    				continue
    			}
    			while(choosing[i] != false);
    			while(ticket[i] != 0 && ticket[i] < ticket[x]);
    			if(ticket[i]==ticket[x] && i<x)
    				while(ticket[i]!=0);
    		}
    		//Sezione critica
    		ticket[x]=0;
    		//Fine sezione critica
    	}
    }
    ```
    

Queste sono soluzioni software, ma si possono implementare anche soluzioni hardware:

- Disabilitare gli interrupt - fare in modo che un thread non ceda il processore finché non finisce. Non funziona sui multiprocessori.
- Istruzione test-and-set - permette al thread di fare le sue operazioni atomicamente. testAndSet(a, b). Si legge il valore di b (booleano), viene copiato in a e infine si setta b a true.
- Istruzione swap - permette di fare uno scambio di variabili atomicamente. swap(a, b)

  

### Semafori

Oggetti java che contengono una variabile il cui contenuto può essere acceduto e modificato da due sole operazioni, P (wait) e V (signal). I semafori inoltre mantengono una coda che memorizza i thread in attesa di accedere alla variabile. Durante l’inizializzazione di un semaforo va specificato il numero di thread che possono accedere contemporaneamente, quindi per avere la mutua esclusione, va settato a 1.

```Java
Semaphore s = new semaphore(1)
```

I semafori possono essere usati per proteggere una sezione critica, ma anche per sincronizzare 2 o più thread. C’è un secondo attributo booleano nel costruttore del semaforo che può essere messo, true se si vuole usare una politica FIFO, false se lo lascio gestire dallo scheduler della virtual machine

Semafori contatori → semafori inizializzati ad un valore più grande di uno, usati quando si deve accedere a più di una risorsa
